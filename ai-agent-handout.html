เอกสารนี้สำหรับคนที่อยากไปต่อ
ทุกศัพท์ที่คลาสจงใจไม่พูด อยู่ที่นี่ครบ

ในห้องเรียนวันนั้น เราจงใจตัดศัพท์และรายละเอียดหลายอย่างออก เพื่อไม่ให้ทุกคนรู้สึกว่า "ยาก" — แต่ถ้าคุณอ่านมาถึงตรงนี้ แปลว่าคุณพร้อมไปต่อแล้ว เอกสารฉบับนี้จะเติมทุกช่องว่างที่คลาสสดจงใจข้าม ตั้งแต่ส่วนประกอบครบทั้ง 8 ตัวของ Agent, ความหมายของ MCP และ Harness แบบละเอียด, ไปจนถึง Agentic AI — แนวคิดการต่อ Agent หลายตัวเป็นทีมงานอัตโนมัติ ซึ่งเป็นก้าวถัดไปหลังจากออกแบบ Agent ตัวแรกได้แล้ว
อ่านต่อจาก: สไลด์สอนสด 60 นาที + ใบงาน Workshop เวลาอ่านโดยประมาณ: 20–25 นาที

2MCP คืออะไร (แบบละเอียด)

ในคลาสสด เราพูดถึง MCP แค่ประโยคเดียวว่า "ปลั๊กมาตรฐานที่ต่อเครื่องมือเข้ากับ AI" แล้วก็ผ่านไป เพราะชื่อเต็มไม่ใช่สิ่งสำคัญที่สุด — สิ่งสำคัญคือเข้าใจว่า MCP ทำหน้าที่อะไรในระบบของ Agent

"ออฟฟิศติดปลั๊กมาตรฐานไว้ทุกโต๊ะ พนักงานคนไหนมาก็เสียบใช้ได้ ไม่ต้องเดินสายใหม่ทุกครั้ง"

ก่อนหน้านี้ ถ้าอยากให้ AI ต่อกับระบบขององค์กร (เช่น ฐานข้อมูลขาย, ปฏิทิน, อีเมล) ทีมพัฒนาต้องเขียนตัวเชื่อมต่อ (connector) ขึ้นใหม่ทุกครั้งที่เปลี่ยน AI ตัวที่ใช้ — MCP คือมาตรฐานกลางที่ทำให้ "ทำ connector ครั้งเดียว AI ตัวไหนก็เสียบใช้ได้" เหมือนที่ USB-C ทำให้อุปกรณ์ต่างยี่ห้อชาร์จสายเดียวกันได้

พูดง่ายๆ: Tools คือ "สิ่งที่ agent ทำได้" (ค้นเว็บ, อ่านไฟล์, ดูปฏิทิน) ส่วน MCP คือ "มาตรฐานของสายที่เสียบ" ที่ทำให้การเพิ่ม Tool ใหม่ๆ ในอนาคตไม่ต้องเริ่มนับหนึ่งใหม่ทุกครั้ง

3Skill vs Tool ต่างกันตรงไหน

สองคำนี้มักถูกเข้าใจปนกัน ทั้งที่ทำหน้าที่คนละอย่างในสมการ Agent = สมอง + มือ + คู่มือ

🖐 Tool = มือ

สิทธิ์เข้าถึงระบบ + อุปกรณ์ที่ใช้ลงมือทำงานจริง

"พนักงานฉลาดแค่ไหน ถ้าไม่มีคอมพิวเตอร์ ไม่มีสิทธิ์เข้าอีเมล ก็ทำได้แค่ตอบจากความจำ"

📘 Skill = คู่มือ

SOP เฉพาะงานที่บอกว่า "งานของเราทำกันแบบไหน"

"เก่งแค่ไหนก็ไม่รู้ว่าสไลด์บริษัทเราใช้ template ไหน — Skill คือคู่มือที่หยิบอ่านเฉพาะตอนทำงานประเภทนั้น"
ToolSkill
ตอบคำถามagent "ทำอะไรได้บ้าง"agent "ทำแบบไหนถึงจะถูกสไตล์เรา"
ตัวอย่างค้นเว็บ, เปิดปฏิทิน, ดึงข้อมูลขาย, คำนวณformat รายงานผู้บริหาร, นิยาม KPI, tone การตอบลูกค้า
ถ้าไม่มีagent ทำได้แค่ตอบจากความรู้เดิม ไม่ลงมือจริงagent ทำงานแบบ freelance ที่เดาสไตล์เอาเอง
จำง่ายๆ: Tool ทำให้ agent "ทำงานได้" ส่วน Skill ทำให้ agent "ทำงานถูกแบบที่องค์กรต้องการ"

4Harness คืออะไร

ถ้า Tool คือมือ และ Skill คือคู่มือ แล้วอะไรที่ทำให้ทุกอย่างทำงานประสานกันจนจบงาน? นั่นคือหน้าที่ของ Harness

"ตัวโปรแกรมที่ห่อทั้งหมดไว้แล้วคุมให้มันทำงานวนจนเสร็จ เขาเรียก Harness — เหมือนตัวออฟฟิศที่พนักงานนั่งทำงาน"

Harness ทำหน้าที่ 4 อย่างหลักๆ:

  • รัน loop "วางแผน → ลงมือทำ → ตรวจผล" จนกว่างานจะเสร็จ
  • คุมสิทธิ์การเข้าถึงระบบและข้อมูลของ agent
  • เก็บ log ทุกการทำงานไว้ย้อนดูได้
  • จัดการเมื่อเกิด error ระหว่างทำงาน
นี่คือเหตุผลว่าทำไม "โมเดลตัวเดียวกัน แต่คนละ Harness ถึงทำงานต่างกันมาก" — สมองอาจฉลาดเท่ากัน แต่ถ้า Harness ไม่ดี agent ก็ทำงานวนซ้ำผิดที่ ไม่รู้จักหยุด หรือไม่มีจุดตรวจสอบตัวเอง

5Agentic AI: ก้าวถัดไปหลัง Agent ตัวแรก

ส่วนนี้คือ "คลาสที่ 2" — เนื้อหาที่จงใจไม่สอนสดวันนั้น เพราะถ้ายังแยก LLM กับ Agent ไม่ออก การโยนเรื่องทีม AI หลายตัวเข้ามาจะยิ่งงง

Agentic AI คือแนวคิดออกแบบระบบงานให้ Agent หลายตัว/หลายบทบาท ทำงานร่วมกันเป็นทีม แทนที่จะให้ Agent ตัวเดียวทำทุกอย่างตั้งแต่ต้นจนจบ

หัวใจของ Agentic AI: 1 Agent = 1 งาน

Agent ที่ดี = งานแคบ + ข้อมูลครบ → ยิ่งงานเล็ก ยิ่งแม่นยำ และคุมง่าย

อย่าสร้าง Agent ตัวเดียวให้ทำทุกอย่าง — แตกเป็นตัวเล็กๆ ที่ทำงานเดียว แล้วให้มันส่งงานต่อกัน นี่คือสิ่งที่เปลี่ยน "AI ที่พอใช้ได้" ให้กลายเป็น "ระบบที่เชื่อถือได้"

แบบที่ควรเลี่ยง

"Credit Assessment Agent" ตัวเดียว

  • ดึงประวัติ + เช็กเอกสาร + ประเมินการจ่าย + เทียบ policy + เขียน memo รวมในตัวเดียว
  • ทดสอบ/แก้ยาก — ผิดตรงไหนไม่รู้
  • ความแม่นต่ำ · เปลี่ยน policy ที = กระทบทั้งตัว

แบบที่ควรทำ

แตกเป็น Agent ย่อย 5 ตัว ทำงานเดียว

  • Memory · Document · Repayment · Policy · Memo
  • แต่ละตัวแม่นยำ · ทดสอบแยกได้ · นำกลับมาใช้ซ้ำได้
  • เปลี่ยน policy → แก้แค่ Policy Agent ตัวเดียว
หลักคิดง่ายๆ: ถ้าอธิบายงานของ Agent ต้องใช้คำว่า "และ" หลายครั้ง → แปลว่าควรแตกเป็นหลายตัว

ตัวอย่างจริง: แตกงาน Topup เป็น Agent ย่อย

งาน "พิจารณาขอวงเงินเพิ่ม (Topup)" ดูเหมือนงานเดียว แต่จริงๆ ประกอบด้วยหลายขั้นที่ "ตัดสินใจ/ประเมิน" คนละเรื่อง — แตกเป็นงานเล็กที่สุดที่ทำได้:

🧠Memory Agent
ลูกค้าคนนี้เคยทำอะไร · ผลครั้งก่อน
📄Document Agent
เอกสารครบไหม · ขาดอะไร
📊Repayment Agent
จ่ายตรงเวลาแค่ไหน → เกรด good/fair/poor
📋Policy Agent
เข้าเกณฑ์ topup ไหม · วงเงินได้เท่าไหร่
📝Credit Memo Agent
รวมผลทุกตัว → เขียน memo ให้เจ้าหน้าที่
👉วิธีหา
ถามว่า "ขั้นตอนที่คนทำ มีกี่สเต็ป" — แต่ละสเต็ปที่ "ตัดสินใจ/ประเมิน" 1 อย่าง = Agent 1 ตัว
เริ่มจากขั้นตอนที่คนทำอยู่แล้ว · แต่ละสเต็ปที่ "ตัดสินใจ/ประเมิน" 1 อย่าง = Agent 1 ตัว · ปิดท้ายด้วยคนอนุมัติเสมอ

จะเริ่มต้นสู่ Agentic AI ยังไง

ไม่ต้องกระโดดไปสร้างทีม Agent 5 ตัวตั้งแต่วันแรก ใช้สมการเดิมที่เรียนมา (Agent = สมอง + มือ + คู่มือ) แล้วขยายทีละขั้น:

  1. เริ่มจาก Agent เดียวที่ทำงานตัวเองได้เสถียรก่อน — ใช้ Agent Canvas (Section 10) ออกแบบให้เสร็จ 1 ตัว
  2. สังเกตว่างานนั้นมีกี่ขั้นตอนที่ "ลักษณะงานต่างกันชัดเจน" เช่น ขั้นดึงข้อมูล ต่างจากขั้นวิเคราะห์ ต่างจากขั้นตรวจสอบ
  3. แยกแต่ละขั้นเป็น Agent ย่อยของตัวเอง — แต่ละตัวมี Goal, Tool, Skill เฉพาะทางของมันเอง ไม่ต้องรู้ทุกอย่างเหมือน Agent ตัวแรก
  4. ต่อผลลัพธ์ของ Agent หนึ่งให้เป็น input ของอีกตัว เหมือนพนักงานส่งงานต่อกันเป็นทอดๆ ในออฟฟิศ
  5. ยังคงจุดอนุมัติของคนไว้ท้ายสุดเสมอ — ไม่ว่าจะมี Agent กี่ตัวต่อกัน กติกาความปลอดภัยไม่เปลี่ยน

3 คำถามก่อนสร้าง Agent (Checklist)

ก่อนลงมือทำ Agent ตัวใดตัวหนึ่ง ถามตัวเอง 3 ข้อนี้ — ใช้ได้กับทุกงาน:

1
งานนี้ "ทำอย่างเดียว" หรือยัง?

ถ้าอธิบายต้องใช้คำว่า "และ" หลายครั้ง → แตกเป็นหลายตัว (1 Agent = 1 งาน)

2
Agent นี้ต้องใช้ "ข้อมูลอะไร"?

ระบุข้อมูลที่ต้องดึง → นั่นคือ Tool (และมาตรฐาน MCP) ที่ต้องเตรียมให้ Agent ตัวนั้น

3
ตรงไหน "คนต้องอนุมัติ"?

Agent แค่แนะนำ · ทุกการตัดสินใจสำคัญต้องผ่านคนอนุมัติเสมอ

แตกงานให้เล็ก → รู้ว่าต้องใช้ข้อมูลอะไร (MCP) → ร้อยเป็น flow → ให้ AI สั่งการ · คนอนุมัติเสมอ
อ้างอิงขั้นสูง · สำหรับคนที่จะออกแบบเชิงระบบ

5D Agent Framework

เมื่อออกแบบ Agent ที่ซับซ้อนขึ้น มีอีกกรอบคิดที่ช่วยไล่ให้ครบทุกมิติ — มองเป็น "ส่วนขยาย" ของสมการหลัก สมอง + มือ + คู่มือ ไม่ใช่โมเดลใหม่ที่ต้องท่องแทน:

Dimensionความหมายตรงกับสมการหลัก
🔍 Detecttrigger / routing — รู้ว่าเมื่อไรควรทำงานสมอง (Goal)
🧭 Decideหลักฐาน / คำแนะนำ — ตัดสินบนข้อมูลสมอง + คู่มือ
⚡ Actขอบเขต — ลงมือ โดยมีคนอนุมัติจุดเสี่ยงมือ + กติกา
📚 Learnจำผล / feedback — ดีขึ้นจากงานที่ผ่านมาสมอง (Memory)
🛡 Governguardrail / citation / fallbackกติกาความปลอดภัย

ในคลาสสดเราไม่สอนกรอบนี้ เพราะจะเพิ่มศัพท์และโมเดลที่ 2 พร้อมกัน — เก็บไว้ใช้ตอนออกแบบระบบจริงเท่านั้น

เหตุผลที่ไม่สอนสดในคลาสแรก: สำหรับคนที่ยังแยก LLM กับ Agent ไม่ออก การโยน "ทีม AI หลายตัว" เข้ามา คือการสอนวิชาที่สองก่อนสอบผ่านวิชาแรก — ถ้าคุณอ่านมาถึงตรงนี้ แปลว่าคุณผ่านวิชาแรกแล้ว

6งานที่เหมาะ / ยังไม่ควรให้ Agent ทำ

เหมาะกับ Agent

  • สรุปรายงานประจำวัน-สัปดาห์
  • หายอดขาย/operation ที่ผิดปกติ
  • สรุปประชุม + ติดตาม action
  • ตรวจเอกสารเบื้องต้น
  • ตอบคำถามจาก knowledge base
  • draft email / report / presentation
  • รวมข้อมูลจากหลายระบบ
  • สร้าง campaign idea / mockup
  • ตรวจ data quality
  • วิเคราะห์ feedback ลูกค้า

ยังไม่ควรให้ทำเองเต็มตัว

  • อนุมัติสินเชื่อ
  • ตัดสินใจ high-impact กับลูกค้า
  • ส่งข้อมูลลับออกนอกองค์กร
  • แก้ production system
  • ตัดสินใจเชิงกฎหมาย/HR โดยไม่มีคนตรวจ
  • ส่งอีเมลภายนอกโดยไม่มี approval
  • ใช้ข้อมูลส่วนบุคคลโดยไม่มี control

7กติกาความปลอดภัย (Guardrail) ตัวอย่างที่ใช้จริง

  • Agent draft email ได้ แต่ส่งเองไม่ได้
  • Agent วิเคราะห์ข้อมูลได้ แต่ต้องแสดง source
  • Agent สรุป recommendation ได้ แต่คนต้องอนุมัติก่อนใช้กับลูกค้า
  • Agent ดึงข้อมูลได้เฉพาะตามสิทธิ์ของผู้ใช้คนนั้น
  • ทุกการทำงานต้องมี log ย้อนดูได้
"Agent ช่วยคิด ช่วยทำ ช่วยเตรียม — แต่เรื่องเสี่ยงสูงต้องมีคนอนุมัติ"

8เริ่มทดลองอย่างปลอดภัย

การเริ่มออกแบบ Agent ไม่จำเป็นต้องใช้ข้อมูลจริงทันที ขั้นแรกให้ทำตามนี้ก่อน:

  • อธิบายงานในระดับ process
  • ใช้ข้อมูลสมมติ
  • หลีกเลี่ยงชื่อลูกค้า
  • หลีกเลี่ยงข้อมูลพนักงาน
  • หลีกเลี่ยงตัวเลขหรือเอกสารภายในที่ยังไม่ได้รับอนุญาต

เริ่มจากระดับ “คลาน → เดิน → วิ่ง” (Crawl / Walk / Run)

ยิ่งงานเสี่ยงสูง ยิ่งต้องเริ่มที่ “คลาน” — ให้ AI ช่วยเตรียม แล้วคนตรวจทุกชิ้น เมื่อพิสูจน์ความแม่นยำแล้วค่อยขยับระดับขึ้น ไม่มีงานไหนต้องเริ่มที่ “วิ่ง” ตั้งแต่วันแรก

🐢 คลาน (Crawl)

AI ช่วยร่าง/สรุปเท่านั้น คนตรวจทุกชิ้นก่อนใช้

🚶 เดิน (Walk)

AI ทำเองเป็นส่วนใหญ่ คนตรวจเฉพาะจุดสำคัญ

🏃 วิ่ง (Run)

AI ทำอัตโนมัติ คนสุ่มตรวจและดูรายงานย้อนหลัง

ตัวอย่างที่ปลอดภัย

"ฉันมีงานที่ต้องสรุปรายงานยอดขายทุกเช้า โดยต้องดึงข้อมูลจากหลายระบบ ช่วยออกแบบ Agent แบบง่ายๆ ให้หน่อย"

ตัวอย่างที่ไม่ควรทำ

"นี่คือรายชื่อลูกค้าและยอดหนี้จริง ช่วยวิเคราะห์ให้หน่อย"

Prompt Template สำหรับเริ่มต้น

ฉันมีงานที่ทำซ้ำบ่อยคือ: [ใส่งานของคุณ] ช่วยออกแบบ AI Agent แบบง่าย ๆ ให้หน่อย โดยตอบเป็น 4 ส่วน: 1. เป้าหมายของ Agent คืออะไร 2. ต้องใช้ข้อมูลหรือเครื่องมืออะไรบ้าง 3. ผลลัพธ์ที่ควรส่งมอบคืออะไร 4. จุดไหนที่คนควรตรวจหรืออนุมัติก่อน
อย่าใส่ข้อมูลลับ ข้อมูลลูกค้า ข้อมูลพนักงาน หรือข้อมูลภายในที่องค์กรไม่ได้อนุญาต ลงใน AI สาธารณะ — ถ้าจะใช้ข้อมูลจริง ต้องใช้เครื่องมือที่องค์กรอนุมัติ และทำตามนโยบายของ IT/Security เท่านั้น

Key Takeaways — ฉบับติดผนัง

  1. LLM ช่วยตอบจากสิ่งที่มี — Agent ใช้เครื่องมือทำงานแล้วกลับมารายงาน
  2. Agent = สมอง + มือ + คู่มือ
  3. ออกแบบ Agent = เขียน Job Description ไม่ใช่เขียนโค้ด
  4. เรื่องเสี่ยง คนต้องอนุมัติก่อนเสมอ
  5. งานที่เหมาะ = ซ้ำ + ใช้ข้อมูล + มีขั้นตอน + ต้องสรุป + มีเครื่องมือให้เรียก
  6. ก้าวแรก: เอางานน่าเบื่อที่สุดของคุณ ไปพิมพ์ถาม AI ภายในสัปดาห์นี้
  7. อย่าใส่ข้อมูลลับหรือข้อมูลลูกค้าลงใน AI สาธารณะ
